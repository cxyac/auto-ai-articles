# 使用 Claude Code：HTML 的不合理有效性

- **作者**: Thariq (@trq212)

## 正文内容

Markdown 已成为智能体与我们沟通的主要文件格式。它简单、便携，有一定的富文本能力，而且便于你编辑。Claude 甚至已经非常擅长在 Markdown 文件中使用 ASCII 制作图表。

但随着智能体变得越来越强大，我感觉 Markdown 已经成为一种限制性格式。我发现很难阅读超过一百行的 Markdown 文件。我想要更丰富的可视化、颜色和图表，而且我想能够轻松分享它们。

我也越来越不自己编辑这些文件，而是将它们用作规格说明、参考文件、头脑风暴输出等。当我确实需要编辑时，我通常会提示 Claude 来编辑它们，这消除了 Markdown 最大的一个优势。

我开始更喜欢使用 HTML 作为输出格式而不是 Markdown，并越来越多地看到 Claude Code 团队中的其他人也在使用这种方式，这就是原因。

（如果你想先看一些示例，可以在这里看到很多：https://thariqs.github.io/html-effectiveness/，但一定要回来看看更多关于为什么的内容）

## 为什么选择 HTML？

### 信息密度

HTML 可以比 Markdown 传达更丰富的信息。它当然可以做简单的文档结构，如标题和格式化，但它还可以表示各种其他信息，例如：

- 使用表格的表格数据
- 使用 CSS 的设计数据
- 使用 SVG 的插图
- 使用 script 标签的代码片段
- 使用带 JavaScript + CSS 的 HTML 元素的交互
- 使用 SVG 和 HTML 的工作流程
- 使用绝对定位和画布的空间数据
- 使用 image 标签的图片

我甚至可以说，几乎没有 Claude 能读取的信息集是你无法用 HTML 相当高效地表示的。这使它成为模型向你传达深入信息以及你审查信息的高效方式。

我发现，在无法做到这一点的情况下，模型可能会在 Markdown 中做一些效率较低的事情，比如 ASCII 图表，或者我最喜欢的，用 Unicode 字符估算颜色，就像这张来自 Claude Code 的截图。

[Claude Code 尝试在 Markdown 中显示颜色]

### 视觉清晰度与阅读便利性

随着 Claude 能够做更复杂的工作，它也在编写越来越大的规格和计划。实际上，我发现我倾向于不真正阅读超过 100 行的 Markdown 文件，我当然也无法让组织中的其他人阅读它。

但 HTML 文档更容易阅读，Claude 可以将结构组织得视觉上更理想，使用标签、插图、链接等。它甚至可以做到移动响应式，让你根据不同的设备形态以不同的方式阅读。

### 分享便利性

Markdown 文件相当难以分享，因为大多数浏览器不能很好地原生渲染它们。你通常必须将它们作为附件添加到电子邮件或消息中。

使用 HTML，只要你上传文件（例如到 S3），你就可以轻松分享链接。你的同事可以在任何他们想要的地方打开它并轻松引用。

如果是 HTML 格式，有人实际阅读你的规格、报告或 PR 说明的可能性要高得多。

### 双向交互

HTML 可以让你与文档交互，例如你可能希望它添加滑块或旋钮来调整设计，或允许你调整算法中的不同选项来看看会发生什么。你也可以要求它让你将这些更改复制到提示中，然后粘贴回 Claude Code。阅读更多关于我的 playground 文章，看看这种双向交互的示例：https://x.com/trq212/status/2017024445244924382

**数据摄取**

为什么要使用 Claude Code 来制作 HTML 文件而不是 ClaudeAI 或 Claude Design 呢？最大的原因之一是 Claude Code 可以摄取的所有上下文。例如，在写这篇文章时，我让 Claude Code 阅读我的代码文件夹，找出我生成的所有 HTML 文件，对它们进行分组和分类，然后制作一个 HTML 文件，用图表表示每种类型。你在本文中看到的图表就是这一过程的直接结果。

除了文件系统，Claude Code 还可以使用你的 MCP（如 Slack、Linear 等）、你的网页浏览器（通过 Chrome 中的 Claude）、你的 Git 历史等找到额外的上下文。

### 它令人愉悦

用 Claude 制作 HTML 文档更有趣，让我感到更有参与感和投入感，而这本身就足够了。

## 如何开始

我有点担心人们会读这篇文章然后把它变成一个 /html 技能之类的东西。虽然那可能有一些价值，但我想强调的是，你不需要做太多就能让 Claude 做到这一点。你可以直接要求它"制作一个 HTML 文件"或"制作一个 HTML artifact"。

诀窍在于知道你想要 artifact 做什么以及你可能如何使用它。随着时间推移，你可能会制作一个技能，但现在我建议从零开始提示，以了解如何在不同情况下使用它。

## 使用案例

为了让这更具体，我为不同的使用案例制作了许多不同的 HTML 文件。你可以在这里查看所有它们：https://thariqs.github.io/html-effectiveness/，但这里是一个概览。

### 规格、规划与探索

HTML 是 Claude 深入研究问题的丰富画布。当我开始处理一个问题时，我期望制作一个 HTML 文件网络，而不是一个简单的 Markdown 计划。例如，我可能会先要求 Claude Code 进行头脑风暴，创建不同选项的一些探索。然后我会要求它更多地扩展其中一个，也许制作模型或代码片段。最后，当我感觉不错时，我会要求它写一个实施计划。当我对计划满意时，我会创建一个新会话并传入所有这些文件让它实施。

在验证时，我也会要求验证智能体阅读这些文件，它将对需要什么有更广泛的上下文。

**示例提示：**

- "我不确定 onboarding 屏幕要朝哪个方向走。生成 6 种明显不同的方法——改变布局、语调和密度——并将它们作为一个 HTML 文件中的网格布局，这样我可以并排比较。用它所做的权衡来标记每一个。"
- "创建一个彻底的实施计划 HTML 文件，一定要制作一些模型，显示数据流，并添加我可能想要审查的重要代码片段。让它易于阅读和消化。"

**使用场景：**
- 探索代码中实现某物的其他方式
- 探索多种视觉设计

### 代码审查与理解

代码在 Markdown 文件中可能难以阅读。但使用 HTML，我们可以渲染差异、注释、流程图、模块等。使用它来理解智能体编写的代码、获取代码审查，或向审查你代码的人解释 PR。我发现这通常比默认的 Github 差异视图效果更好，我现在每个 PR 都会附上一个 HTML 代码解释器。

**示例提示：**

"通过创建一个 HTML artifact 来帮助我审查这个 PR。我对流式/背压逻辑不太熟悉，所以重点关注那个。用内联边距注释渲染实际的差异，按严重性对发现进行颜色编码，以及传达概念良好所需的任何其他内容。"

**使用场景：**
- 创建 PR
- 审查 PR
- 理解代码中的某个主题

### 设计与原型

Claude Design 基于 HTML，因为 HTML 在设计方面极具表现力，即使你的最终界面不是 HTML。Claude 可以用 HTML 勾勒出设计，然后用你选择的语言编写，无论是 React、Swift 等。

你也可以原型化交互，如动画、动作等。考虑要求 Claude 制作滑块、旋钮等，以精确调整你要找的东西。

**示例提示：**

"我想原型化一个新的结账按钮，点击时播放动画然后快速变成紫色。创建一个带有几个滑块和选项的 HTML 文件，让我尝试这个动画的不同选项，给我一个复制按钮来复制效果良好的参数。"

**用于：**
- 创建设计系统 artifacts
- 调整组件
- 可视化组件库
- 原型化有趣的动画

### 报告、研究与学习

Claude Code 非常擅长综合多个数据源的信息并将其转换为可读性强的报告。你可以提示 Claude 搜索你的 Slack、代码库、Git 历史、互联网等，并为你自己、领导层、团队等生成极其可读的报告。

你可以将其组装成一个长 HTML 文档、一个交互式解释器，甚至一个幻灯片/演示文稿。要求 Claude 使用 SVG 制作图表来帮助可视化。例如，对于我的关于提示缓存的帖子，我让 Claude 在阅读 Git 历史后，为我准备了一个关于我们对提示缓存的所有更改的深入研究 HTML 文件。

**示例提示：**

"我不理解我们的速率限制器实际是如何工作的。阅读相关代码并生成一个单一的 HTML 解释页面：令牌桶流程的图表、3-4 个带注释的关键代码片段，以及底部的'陷阱'部分。为只读一次的人优化。"

**用于：**
- 总结某个功能如何工作
- 向我解释一个概念
- 每周向你的老板报告状态
- 向你的领导层报告事件
- SVG 插图、流程图、技术图表等

## 自定义编辑界面

有时很难在纯文本框中描述你想要什么。在这种情况下，我会要求 Claude 为我正在处理的精确事物构建一个一次性编辑器。不是一个产品，也不是一个可重用的工具，而是一个单一的 HTML 文件，为这个特定的数据专门构建。

诀窍总是以导出结束：一个"复制为 JSON"或"复制为提示"按钮，将我在 UI 中所做的任何操作转换回可以粘贴到 Claude Code 中的内容。

**示例提示：**

- "我需要重新确定这 30 个 Linear 工单的优先级。给我制作一个 HTML 文件，每个工单作为一个可拖动的卡片，跨 Now / Next / Later / Cut 列。按你的最佳猜测预排序。添加一个'复制为 Markdown'按钮，导出最终排序，每个桶有一行理由。"
- "这是我们的功能标志配置。为它构建一个基于表单的编辑器，按区域分组标志，显示它们之间的依赖关系，如果我启用一个先决条件关闭的标志，警告我。添加一个'复制差异'按钮，只给我改变的键。"
- "我正在调整这个系统提示。制作一个并排编辑器：左侧可编辑的提示，变量插槽高亮显示，右侧三个样本输入，实时重新渲染填充的模板。添加字符/令牌计数器和复制按钮。"

**用于：**
- 重新排序、分类或分桶任何内容（工单、测试用例、反馈）
- 编辑结构化配置（功能标志、环境变量、带约束的 JSON/YAML）
- 调整提示、模板或文案，带实时预览
- 策划数据集、批准/拒绝行、标记示例、导出的选择
- 注释文档、转录本或差异并导出注释
- "选择用文本表达痛苦的值：颜色、缓动曲线、裁剪区域、cron 计划、正则表达式。"

## 常见问题

我告诉了很多人关于我如何转向 HTML，看到了一些重复的问题。

**Q: 它的令牌效率不是更低吗？**

虽然 Markdown 通常使用更少的令牌，但我发现 HTML 的增加表现力以及我阅读它的可能性大大提高意味着我获得更好的整体输出。在 Opus 4.7 的 1MM 上下文窗口中，增加的令牌使用在上下文窗口中并不明显。

**Q: 你现在什么时候用 Markdown？**

老实说，我几乎已经停止使用 Markdown 来做几乎所有事情了，但我可能是 HTML 最大化主义者那边的人。

**Q: 如何查看 HTML 文件？**

我倾向于只是在本地浏览器中打开它（你可以要求 Claude 打开它），或者如果你想有一个可分享的链接，上传到 S3。

**Q: 这比 Markdown 生成长吗？**

这确实需要更长的时间！HTML 可能比 Markdown 花费 2-4 倍的时间，但我发现结果是值得的。

**Q: 版本控制呢？**

这 honestly 是 HTML 最大的缺点之一，HTML 差异嘈杂且难以审查，相比 Markdown。

**Q: 如何让 Claude 符合我的品味/不让它变得难看？**

前端设计插件帮助 Claude 制作良好的 HTML 文件。但要符合你自己公司的风格，你可以通过指向你的代码库让 Claude 创建一个单一的设计系统 HTML 文件。然后你可以将该设计系统文件用作其他 HTML 文件的参考。

## 保持同步

以上所有内容都是说，我认为我使用 HTML 的真正原因是我感觉与 Claude 更加同步。我曾开始担心，因为我已经停止深入阅读计划，我将不得不让 Claude 做出它的选择。

但我很高兴地说，相反，当使用 HTML 时，我感到比以往任何时候都更加同步。我希望你也是如此。
